## Slide 7 — 功能特色①：安全守护闭环

**页面标题：** 功能特色 ①

**页面副标题：** 安全守护闭环 — 从传感器报警到QQ推送，只需1秒

**核心话术：**
- 第一个特色是安全守护闭环。这不是一个模拟Demo，而是从真实硬件到用户手机的端到端闭环。
- 硬件端使用 BearPi-HM Nano 开发板搭载 MQ2 烟雾传感器，每秒采集 ppm 值，通过 MQTT 上报到云端。

- 告警通过 Spring 事件机制解耦推送，经过 NapCat QQ 机器人，秒级推送到用户手机。
- 同时，系统会自动下发指令联动蜂鸣器、排风扇和报警灯。

## Slide 12 — 端侧：硬件与固件

**页面标题：** 端侧：硬件与固件

**页面副标题：** BearPi-HM Nano 上的非阻塞式告警状态机

**核心话术：**
- 硬件端我们选用 BearPi-HM Nano 开发板，搭载 Hi3861 WiFi 芯片和 E53_SF1 扩展板。
- 传感器方面，MQ2 烟雾传感器接在 ADC 通道6，每秒读取一次 ppm 值。
- 执行器包括蜂鸣器、LED报警灯和排风扇，通过 GPIO 控制。
- 固件的核心是一个**非阻塞式告警状态机**，每100毫秒执行一次 tick，根据 ppm 阈值自动切换正常/告警状态，同时支持 F1 按键强制告警、F2 按键一键恢复。
- 这种设计保证了即使 MQTT 网络波动，本地硬件也能独立做出安全决策。

## Slide 13 — 边侧：MQTT设备微服务

**页面标题：** 边侧：MQTT设备微服务

**页面副标题：** 4个独立微服务，解耦数据采集与控制转发

**核心话术：**
- 设备端我们拆分了4个独立部署的 Java 微服务，全部基于 Spring Boot 2.3.5。
- getData 服务订阅 group23 topic，解析 JSON 中的 ppm、温度、湿度，分别写入三张传感器数据表。它内置了风险分级计算器。
- postData 服务每秒轮询 device_control 数据表，一旦发现状态变更，就通过 MQTT 下发到 group23-s-to-h topic，同步控制硬件端的蜂鸣器、LED和开关。
- simulate 服务是一个正态分布温湿度模拟器，用于无硬件时的开发和演示。
- MQTT工具服务提供 REST API，方便调试时手动发布消息。
- 这些微服务都覆盖了单元测试，getData 还包含远程 MQTT 集成测试。

## Slide 14 — 云端：后端与数据

**页面标题：** 云端：后端与数据

**页面副标题：** Spring Boot 3 + 21张数据表 + 混合数据隔离策略

**核心话术：**
- 主后端采用 Spring Boot 3.3.5，Java 17，共15个 Controller，63个 HTTP 端点。
- 数据库使用 MySQL 8，设计了21张表加1个视图，覆盖用户、宠物、传感器、设备、告警、行为记录、账单等全业务域。
- 这里有一个值得分享的设计决策：**混合数据隔离策略**。
- 宠物档案、医疗记录、个人偏好等数据是用户级的，通过 ThreadLocal 的 UserContext 实现隔离；而传感器数据、设备状态、告警记录是全局共享的，因为物理上只有一个传感器。这种设计既保证了用户隐私，又避免了数据冗余。
- 安全方面，用户可以在设置页自行配置 Qwen 和 DeepSeek 的 API Key，后端使用 AES-256-GCM 加密存储，读取时脱敏显示，且优先级高于配置文件，无需重启服务。

**视觉建议：**
- 左侧：后端技术栈图标墙（Spring Boot / JPA / MySQL / Redis / Validation / Lombok）
- 右侧：两张小架构图
  - 图1：混合数据隔离示意图（用户数据 vs 全局数据）
  - 图2：AES-256-GCM 加密流程（明文 → 加密 → ENC:前缀存储 → 脱敏显示）

## Slide 15 — AI技术栈：三模态融合

**页面标题：** AI技术栈：三模态融合

**页面副标题：** 本地离线 + 云端多模态 + LLM Agent 协同工作

**核心话术：**
- 我们的 AI 系统不是单一的，而是三套模型协同。
- **本地离线层**：YOLOv8n ONNX 负责检测鹦鹉位置，CLIP 负责零样本行为和品种分类。模型文件就放在 smartjavaai-models 目录，开箱即用，不依赖网络。
- **云端多模态层**：Qwen-VL 通义千问视觉大模型，用于 3D 虚拟笼场景的理解和问答。API Key 由用户自行配置。
- **LLM Agent 层**：DeepSeek 的 Function Calling 能力驱动 QQ Bot，理解自然语言意图并调用工具。
- 这三层各司其职：本地模型保证隐私和实时性，云端模型提供复杂理解能力，Agent 层负责交互决策。

**视觉建议：**
- 一张**三模态金字塔图**：
  - 底层（宽）：YOLOv8n + CLIP — 本地离线 — 实时检测
  - 中层：Qwen-VL — 云端API — 多模态理解
  - 顶层（尖）：DeepSeek — 云端API — 交互决策

## Slide 16 — QQ Bot Agent 架构

**页面标题：** QQ Bot Agent 架构

**页面副标题：** 事件驱动 × 三层降级 × 安全确认

**核心话术：**
- QQ Bot 是我们最复杂也最具创新的模块，我用三个关键词概括它。
- 第一，**事件驱动**。烟雾告警不是轮询查出来的，而是通过 Spring 的 AlarmTriggeredEvent 事件机制解耦触发的。业务代码只管发事件，推送逻辑独立消费，实现了真正的解耦。
- 第二，**三层降级**。当用户问问题时，优先走 DeepSeek LLM；如果模型超时或不可用， fallback 到规则匹配（比如"开蜂鸣器"直接映射指令）；如果连规则也不匹配，最后走 MaxKB 知识库问答。
- 第三，**安全确认**。设备控制类指令需要60秒内二次确认，Redis 存储 pending 状态，防止误触。QQ 白名单也支持可视化管理和数据库优先配置。

**视觉建议：**
- 一张**事件驱动架构图**：
  - 左侧：AlarmService 发布 AlarmTriggeredEvent
  - 中间：Spring EventBus
  - 右侧：OneBotPushService 监听 → 调用 NapCat → QQ消息
- 下方并列**三层降级流程图**：LLM → 规则 → MaxKB，用箭头标注 fallback 方向
- 底部标注：Redis 60s pending-op 二次确认

- 第五，**3D 沉浸式交互**。three.js 虚拟鹦鹉笼超越了传统仪表盘的数据展示逻辑。
