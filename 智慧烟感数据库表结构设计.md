# 智慧宠物烟感安全系统 - 数据库表结构设计

> 版本：v2.0
> 日期：2026-07-05
> 数据库：MySQL 5.7+
> 字符集：utf8mb4
> 引擎：InnoDB

---

## 0. 设计概览

本设计在原有智慧烟感核心表基础上，围绕「宠物笼舍环境监测 + 烟感安全告警 + 宠物成长照护」三条主线进行扩展。所有表统一使用 `utf8mb4` 字符集，表之间采用逻辑关联，不创建数据库外键。

| 序号 | 表名 | 中文名 | 说明 | 预估数据量 |
|------|------|--------|------|-----------|
| 1 | `sys_user` | 用户表 | 登录账号与权限 | 少量（<100） |
| 2 | `user_preference` | 用户偏好表 | 主题、字体、通知、语言等个性化配置 | 少量 |
| 3 | `pet_cage` | 宠物笼舍表 | 笼舍/监测区域信息，绑定监测设备 | 少量（<1000） |
| 4 | `smoke_device` | 烟感设备表 | 设备基本信息与当前状态 | 少量（<1000） |
| 5 | `device_control` | 联动设备表 | 蜂鸣器/报警灯/排风扇/空气净化器等 | 少量 |
| 6 | `system_setting` | 系统设置表 | 阈值、全局配置 | 极少（KV形式） |
| 7 | `smoke_data` | 烟雾/粉尘数据表 | 历史浓度数据，量最大 | 海量（按采样频率） |
| 8 | `temperature_data` | 温度数据表 | 环境温度历史数据 | 海量（按采样频率） |
| 9 | `humidity_data` | 湿度数据表 | 环境相对湿度历史数据 | 海量（按采样频率） |
| 10 | `alarm_record` | 告警记录表 | 告警事件主表 | 中等 |

## 1. 用户表 `sys_user`

用于系统登录与用户管理。前端「用户设置」模块展示当前账号信息。

```sql
CREATE TABLE `sys_user` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `username` VARCHAR(64) NOT NULL COMMENT '用户名',
  `password` VARCHAR(128) NOT NULL COMMENT '加密密码（建议 bcrypt）',
  `real_name` VARCHAR(64) DEFAULT NULL COMMENT '真实姓名',
  `role` VARCHAR(32) NOT NULL DEFAULT 'viewer' COMMENT '角色：admin-管理员, viewer-只读用户',
  `phone` VARCHAR(20) DEFAULT NULL COMMENT '手机号',
  `email` VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
  `avatar_url` VARCHAR(500) DEFAULT NULL COMMENT '头像URL',
  `location` VARCHAR(255) DEFAULT NULL COMMENT '用户位置信息（如 上海市·浦东新区）',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1-启用, 0-禁用',
  `last_login_time` DATETIME DEFAULT NULL COMMENT '最后登录时间',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`),
  KEY `idx_role` (`role`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

## 3. 宠物笼舍表 `pet_cage`

```sql
CREATE TABLE `pet_cage` (

  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_cage_id` (`cage_id`),
  KEY `idx_device_id` (`device_id`),
```

## 4. 烟感设备表 `smoke_device`

核心设备表。原型中「设备编号: SMOKE-001」「设备在线 12/12」均来自此表。在宠物场景下，设备负责监测笼舍内烟雾/粉尘浓度，守护宠物呼吸安全。

```sql
CREATE TABLE `smoke_device` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `device_id` VARCHAR(64) NOT NULL COMMENT '设备编号（业务唯一，如 SMK-001）',
  `name` VARCHAR(128) NOT NULL COMMENT '设备名称（如 笼舍A01烟感）',
  `cage_id` VARCHAR(64) DEFAULT NULL COMMENT '逻辑关联 pet_cage.cage_id',
  `location` VARCHAR(255) DEFAULT NULL COMMENT '安装位置',
  `online` TINYINT NOT NULL DEFAULT 1 COMMENT '在线状态：1-在线, 0-离线',
  `last_heartbeat` DATETIME DEFAULT NULL COMMENT '最近心跳时间',
  `current_smoke_value` INT DEFAULT 0 COMMENT '当前烟雾/粉尘浓度（ppm，冗余缓存最新值）',
  `current_risk_level` VARCHAR(32) DEFAULT 'normal' COMMENT '当前风险等级：normal/low/medium/high',
  `current_alarm_status` VARCHAR(32) DEFAULT 'safe' COMMENT '当前报警状态：safe/alarm/offline',
  `enabled` TINYINT NOT NULL DEFAULT 1 COMMENT '是否启用：1-启用, 0-禁用',
  `remark` VARCHAR(500) DEFAULT NULL COMMENT '备注',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_device_id` (`device_id`),
  KEY `idx_cage_id` (`cage_id`),
  KEY `idx_online` (`online`),
  KEY `idx_enabled` (`enabled`),
  KEY `idx_alarm_status` (`current_alarm_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='烟感设备表';
```

**设计要点**

- `current_smoke_value`、`current_risk_level`、`current_alarm_status` 是**冗余字段**，用于首页快速查询，避免每次查历史数据表。由后端在收到 MQTT 数据或 `/api/sensor/upload` 时更新。
- `online` 状态由心跳检测服务定时更新（如 5 分钟未收到心跳则置为离线）。
- `cage_id` 用于将设备与宠物笼舍绑定，前端「实时监控」按笼舍维度聚合展示。

## 5. 联动设备表 `device_control`

蜂鸣器、报警灯、排风扇、空气净化器等联动控制设备的状态。原型右侧「设备控制」面板，在宠物场景下用于告警时自动排除有害气体或提醒主人。

```sql
CREATE TABLE `device_control` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `device_id` VARCHAR(64) NOT NULL COMMENT '所属烟感设备编号',
  `control_type` VARCHAR(64) NOT NULL COMMENT '控制设备类型：buzzer-蜂鸣器, alarm_light-报警灯, fan-排风扇, air_purifier-空气净化器',
  `control_name` VARCHAR(128) DEFAULT NULL COMMENT '设备显示名称（如 蜂鸣器）',
  `status` VARCHAR(32) NOT NULL DEFAULT 'off' COMMENT '当前状态：on-开启, off-关闭',
  `auto_linkage` TINYINT NOT NULL DEFAULT 1 COMMENT '是否参与自动联动：1-是, 0-否',
  `last_operated_at` DATETIME DEFAULT NULL COMMENT '最后操作时间',
  `last_operated_by` VARCHAR(64) DEFAULT NULL COMMENT '最后操作人（system 表示系统自动）',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_device_control` (`device_id`, `control_type`),
  KEY `idx_device_id` (`device_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='联动控制设备表';
```

**初始化数据**

```sql
INSERT INTO `device_control` (`device_id`, `control_type`, `control_name`, `status`) VALUES
('SMK-001', 'buzzer', '蜂鸣器', 'off'),
('SMK-001', 'alarm_light', '报警灯', 'off'),
('SMK-001', 'fan', '排风扇', 'off'),
('SMK-001', 'air_purifier', '空气净化器', 'off');
```

**设计要点**

- 一个烟感设备 (`smoke_device`) 对应多个联动设备 (`device_control`)，1:N 关系。
- `auto_linkage` 用于标识该设备是否在告警时自动联动开启。
- 危险状态自动联动逻辑：当 `alarm_record` 插入 medium/high 级别告警时，后端自动将所有 `auto_linkage=1` 的设备 status 改为 `on`，并写入操作记录。

## 7. 烟雾/粉尘数据表 `smoke_data`

历史浓度数据，量最大的表。用于「烟雾/粉尘浓度历史趋势」折线图。在宠物场景下，烟雾传感器同时用于监测笼舍羽粉/粉尘浓度，保护宠物呼吸道健康。

```sql
CREATE TABLE `smoke_data` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `device_id` VARCHAR(64) NOT NULL COMMENT '设备编号',
  `smoke_value` FLOAT NOT NULL COMMENT '烟雾/粉尘浓度（ppm）',
  `risk_level` VARCHAR(32) NOT NULL COMMENT '风险等级：normal/low/medium/high',
  `record_time` DATETIME NOT NULL COMMENT '数据记录时间（传感器上报时间）',
  `source` VARCHAR(32) DEFAULT 'sensor' COMMENT '数据来源：sensor-真实传感器, simulate-模拟',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '入库时间',
  PRIMARY KEY (`id`),
  KEY `idx_device_time` (`device_id`, `record_time`),
  KEY `idx_record_time` (`record_time`),
  KEY `idx_risk_level` (`risk_level`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='烟雾/粉尘历史数据表';
```

**设计要点**

| 要点 | 说明 |
|------|------|
| 索引策略 | `idx_device_time` 覆盖 99% 的查询（按设备+时间范围查历史） |
| 分区建议 | 数据量大时按 `record_time` 按月分区，或定时归档到 `smoke_data_archive` |
| 聚合查询 | 首页 24h/7d 趋势不要直接读原始数据，建议定时任务预聚合到 `smoke_data_hourly` |
| 数据保留 | 原始数据保留 30 天，聚合数据保留 1 年 |

**预聚合表（可选，推荐）**

```sql
CREATE TABLE `smoke_data_hourly` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `device_id` VARCHAR(64) NOT NULL,
  `hour` DATETIME NOT NULL COMMENT '整点小时（如 2026-07-05 14:00:00）',
  `avg_value` INT NOT NULL COMMENT '小时平均浓度',
  `max_value` INT NOT NULL COMMENT '小时最大浓度',
  `min_value` INT NOT NULL COMMENT '小时最小浓度',
  `sample_count` INT NOT NULL COMMENT '采样点数',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_device_hour` (`device_id`, `hour`),
  KEY `idx_hour` (`hour`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='烟雾/粉尘浓度小时聚合表';
```

记录设备上报的环境相对湿度，用于实时监控页面的湿度展示和后续历史趋势分析。

```sql
CREATE TABLE `humidity_data` (

  `device_id` VARCHAR(64) NOT NULL COMMENT '设备编号',
  `humidity_value` FLOAT NOT NULL COMMENT '相对湿度（%RH）',
  `record_time` DATETIME NOT NULL COMMENT '数据记录时间（传感器上报时间）',
  `source` VARCHAR(32) DEFAULT 'sensor' COMMENT '数据来源：sensor-真实传感器, simulate-模拟',

  PRIMARY KEY (`id`),
  KEY `idx_device_time` (`device_id`, `record_time`),
  KEY `idx_record_time` (`record_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='湿度历史数据表';
```

**温湿度表设计要点**

- 与 `smoke_data` 使用相同的设备编号、采集时间、数据来源和组合索引模式。
- 温湿度没有烟雾风险等级，因此不设置 `risk_level`。
- 两张表用于保存原始采样数据，建议同样保留 30 天；需要长期趋势时可按小时预聚合。
- 宠物场景下，温湿度结合 `system_setting` 中的舒适区间，前端展示「偏低/适宜/偏高」。

## 10. 告警记录表 `alarm_record`

告警事件主表。原型中「历史报警记录」「今日告警」均来自此表。宠物场景下，告警类型扩展为烟雾/粉尘超标、温湿度异常、设备离线等。

```sql
CREATE TABLE `alarm_record` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `alarm_id` VARCHAR(64) NOT NULL COMMENT '告警编号（业务唯一，如 ALM-20260705-001）',
  `device_id` VARCHAR(64) NOT NULL COMMENT '设备编号',
  `cage_id` VARCHAR(64) DEFAULT NULL COMMENT '逻辑关联 pet_cage.cage_id',
  `alarm_type` VARCHAR(64) NOT NULL COMMENT '告警类型：smoke_high-烟雾/粉尘浓度超限, temp_high-温度过高, temp_low-温度过低, humidity_high-湿度过高, humidity_low-湿度过低, device_offline-设备离线',
  `smoke_value` INT DEFAULT NULL COMMENT '触发时浓度（ppm，非烟雾告警为null）',
  `risk_level` VARCHAR(32) NOT NULL COMMENT '风险等级：low/medium/high',
  `status` VARCHAR(32) NOT NULL DEFAULT 'pending' COMMENT '处理状态：pending-待处理, processing-处理中, resolved-已处理',
  `handler` VARCHAR(64) DEFAULT NULL COMMENT '处理人用户名',
  `handled_at` DATETIME DEFAULT NULL COMMENT '处理时间',
  `remark` VARCHAR(1000) DEFAULT NULL COMMENT '处理备注',
  `triggered_at` DATETIME NOT NULL COMMENT '告警触发时间',
  `resolved_at` DATETIME DEFAULT NULL COMMENT '告警恢复时间（手动恢复或自动恢复）',
  `is_simulated` TINYINT NOT NULL DEFAULT 0 COMMENT '是否模拟触发：1-是, 0-否',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_alarm_id` (`alarm_id`),
  KEY `idx_device_id` (`device_id`),
  KEY `idx_cage_id` (`cage_id`),
  KEY `idx_status` (`status`),
  KEY `idx_triggered_at` (`triggered_at`),
  KEY `idx_alarm_type` (`alarm_type`),
  KEY `idx_composite` (`device_id`, `status`, `triggered_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警记录表';
```

**设计要点**

- `alarm_id` 生成规则建议：`ALM-YYYYMMDD-NNN`（如 `ALM-20260705-001`），也可用雪花算法。
- `status` 流转：pending → processing → resolved（单向，不可回退）。
- `is_simulated` 用于标记「模拟烟雾升高」触发的告警，便于演示后清理或统计区分。
- `cage_id` 方便前端按笼舍筛选告警，快速定位受影响的宠物。
- 今日告警统计 SQL 示例：

```sql
-- 今日告警次数（中风险及以上）
SELECT COUNT(*) FROM alarm_record 
WHERE triggered_at >= CURDATE() 
  AND triggered_at < DATE_ADD(CURDATE(), INTERVAL 1 DAY)
  AND risk_level IN ('medium', 'high');

-- 较昨日变化
SELECT 
  (SELECT COUNT(*) FROM alarm_record WHERE triggered_at >= CURDATE() AND risk_level IN ('medium','high')) AS today_count,
  (SELECT COUNT(*) FROM alarm_record WHERE triggered_at >= DATE_SUB(CURDATE(),INTERVAL 1 DAY) AND triggered_at < CURDATE() AND risk_level IN ('medium','high')) AS yesterday_count;
```

```
                          │    setting_group│
                          └─────────────────┘

                          ┌─────────────────┐
                          │food_safety_query│
                          ├─────────────────┤
                          │ PK id           │
                          │ FK user_id      │
                          │    food_name    │
                          │    result       │
                          │    queried_at   │
                          └─────────────────┘
```

关系说明：`smoke_device.device_id` 与 `smoke_data`、`temperature_data`、`humidity_data`、`alarm_record`、`device_control` 的 `device_id` 形成逻辑上的 1:N 关系；`pet_cage.cage_id` 1:1 `smoke_device.cage_id`，1:N `pet_profile.cage_id` / `alarm_record.cage_id` / `pet_media_record.cage_id`；`sys_user.id` 1:N `pet_profile.user_id` / `user_preference.user_id` / `pet_ledger_record.user_id` / `food_safety_query.user_id`，`pet_profile.pet_id` 1:N `pet_weight_record.pet_id` / `pet_daily_report.pet_id` / `pet_media_record.pet_id` / `pet_medical_record.pet_id` / `pet_ledger_record.pet_id`。当前统一使用逻辑关联，不创建数据库外键。

## 21. 核心业务流程与 SQL 示例

### 21.1 首页总览数据查询

```sql
-- 1. 当前宠物关联设备的最新烟雾/粉尘浓度（取最新一条）
SELECT smoke_value, risk_level, record_time 
FROM smoke_data 
WHERE device_id = 'SMK-001' 
ORDER BY record_time DESC LIMIT 1;

-- 2. 当前温度和湿度（各取最新一条）
SELECT temperature_value, record_time
FROM temperature_data
WHERE device_id = 'SMK-001'
ORDER BY record_time DESC LIMIT 1;

SELECT humidity_value, record_time
FROM humidity_data
WHERE device_id = 'SMK-001'
ORDER BY record_time DESC LIMIT 1;

-- 3. 今日告警统计
SELECT 
  COUNT(*) AS today_count,
  SUM(CASE WHEN alarm_type = 'device_offline' THEN 1 ELSE 0 END) AS offline_count
FROM alarm_record 
WHERE triggered_at >= CURDATE() 
  AND triggered_at < DATE_ADD(CURDATE(), INTERVAL 1 DAY)
  AND risk_level IN ('medium', 'high');

-- 4. 设备在线数
SELECT 
  SUM(online = 1) AS online_count,
  COUNT(*) AS total_count
FROM smoke_device;

-- 5. 最近5条告警（首页表格）
SELECT alarm_id, triggered_at AS alarm_time, device_id, cage_id, alarm_type, 
       smoke_value, risk_level AS level, status
FROM alarm_record 
ORDER BY triggered_at DESC 
LIMIT 5;

-- 6. 当前登录用户偏好
SELECT pref_key, pref_value, pref_group
FROM user_preference
WHERE user_id = 1;
```

### 21.2 模拟烟雾/粉尘升高（后端事务）

```sql
-- 步骤1：插入高浓度数据
INSERT INTO smoke_data (device_id, smoke_value, risk_level, record_time, source)
VALUES ('SMK-001', 520, 'high', NOW(), 'simulate');

-- 步骤2：更新设备当前状态
UPDATE smoke_device 
SET current_smoke_value = 520, current_risk_level = 'high', current_alarm_status = 'alarm', updated_at = NOW()
WHERE device_id = 'SMK-001';

-- 步骤3：生成告警记录
INSERT INTO alarm_record (alarm_id, device_id, cage_id, alarm_type, smoke_value, risk_level, status, triggered_at, is_simulated)
VALUES ('ALM-20260705-001', 'SMK-001', 'CAGE-A01', 'smoke_high', 520, 'high', 'pending', NOW(), 1);

-- 步骤4：写入时间线
INSERT INTO alarm_timeline (alarm_id, event_type, event_desc, operator, event_time)
VALUES ('ALM-20260705-001', 'trigger', '告警触发：粉尘浓度 520 ppm', 'system', NOW());

-- 步骤5：自动联动开启设备
UPDATE device_control SET status = 'on', last_operated_at = NOW(), last_operated_by = 'system'
WHERE device_id = 'SMK-001' AND auto_linkage = 1;

-- 步骤6：记录联动事件
INSERT INTO alarm_timeline (alarm_id, event_type, event_desc, operator, extra_data, event_time)
VALUES ('ALM-20260705-001', 'linkage_start', '设备联动启动', 'system', '{"devices":["buzzer","alarm_light","fan","air_purifier"]}', NOW());
```

### 21.3 恢复正常环境（后端事务）

```sql
-- 步骤1：插入正常浓度数据
INSERT INTO smoke_data (device_id, smoke_value, risk_level, record_time, source)
VALUES ('SMK-001', 35, 'normal', NOW(), 'simulate');

-- 步骤2：更新设备状态
UPDATE smoke_device 
SET current_smoke_value = 35, current_risk_level = 'normal', current_alarm_status = 'safe', updated_at = NOW()
WHERE device_id = 'SMK-001';

-- 步骤3：关闭联动设备
UPDATE device_control SET status = 'off', last_operated_at = NOW(), last_operated_by = 'system'
WHERE device_id = 'SMK-001';

-- 步骤4：将当前 pending/processing 告警置为 resolved
UPDATE alarm_record 
SET status = 'resolved', resolved_at = NOW(), updated_at = NOW()
WHERE device_id = 'SMK-001' AND status IN ('pending', 'processing');
```

### 21.4 按笼舍查询宠物及环境快照

```sql
-- 查询某笼舍下的宠物档案与设备最新状态
SELECT 
  p.pet_id, p.name, p.species, p.weight_grams, p.current_status,
  d.device_id, d.online, d.current_smoke_value, d.current_risk_level
FROM pet_profile p
LEFT JOIN pet_cage c ON p.cage_id = c.cage_id
LEFT JOIN smoke_device d ON c.device_id = d.device_id
WHERE p.user_id = 1 AND p.enabled = 1;
```

### 21.5 成长报告日报聚合

```sql
-- 按宠物和日期查询成长日报
SELECT * FROM pet_daily_report
WHERE pet_id = 'PET-001' AND report_date BETWEEN '2026-06-01' AND '2026-07-05'
ORDER BY report_date DESC;

-- 周报聚合示例
SELECT 
  YEAR(report_date) AS year,
  WEEK(report_date) AS week,
  AVG(health_score) AS avg_health_score,
  SUM(sleep_minutes) AS total_sleep_minutes,
  AVG(dust_avg_value) AS avg_dust
FROM pet_daily_report
WHERE pet_id = 'PET-001'
GROUP BY YEAR(report_date), WEEK(report_date)
ORDER BY year DESC, week DESC;
```

## 22. 性能优化建议

| 场景 | 优化方案 |
|------|----------|
| `smoke_data` 数据量大 | 按 `record_time` 做 Range 分区，或按月分表 |
| 温湿度历史数据量大 | `temperature_data`、`humidity_data` 按 `record_time` 归档或按月分区 |
| 首页历史趋势查询慢 | 走 `smoke_data_hourly` 聚合表，不直接查原始数据 |
| 设备心跳频繁更新 | `last_heartbeat` 更新可用缓存+批量写，减少单条 UPDATE |
| 今日告警统计 | 增加 Redis 计数器，由后端定时任务校准，避免实时 COUNT(*) |
| 告警列表分页 | `idx_triggered_at` + 倒序分页，深分页用游标/延迟关联 |
| 宠物记账查询 | 使用 `idx_pet_expense_date` 按宠物和日期范围查询、汇总 |
| 宠物媒体记录 | 按 `pet_id` + `media_type` 分表或归档到对象存储，表内只保留元数据 |
| 成长报告查询 | 使用 `idx_pet_report_date` 按宠物和日期范围查询 |

## 23. 扩展建议

1. **多宠家庭扩展**：当前设计已支持多宠物、多笼舍，若后续有「家庭/房间」层级，可增加 `family` / `room` 表。
2. **操作日志**：增加 `operation_log` 表记录所有人工操作（谁、什么时候、操作了什么设备/告警/宠物档案），用于审计。
3. **消息通知**：增加 `notification` 表记录推送通知（APP推送、短信、邮件），支持重试。
4. **定时任务表**：若用轻量级调度，可增加 `scheduled_task` 表管理数据清理、聚合任务。
5. **智能体对话历史**：增加 `chat_history` 表保存智能问答上下文，支持多轮对话与知识库优化。
6. **饮食计划**：增加 `pet_feeding_plan` 表管理喂养计划与提醒。
7. **疫苗/驱虫提醒**：增加 `pet_care_reminder` 表管理疫苗、驱虫、体检等周期性提醒。
